BICMD database structures - reading/writing

reading

writing

  • the SQL structure is used as source message

  • it’s possible to automatically update the table

  • it’s possible to set read conditions

  • the used tables are linked by the primary and foreign keys

  • the SQL structure is used as destination

  • mapping works like an INHOUSE mapping

  • the foreign key information of the related table are filled automatically

  • there exist different writing modes (BICSQLMODE)

    • INSERT
    • INSERT_OR_UPDATE
    • INSERT_DELETING
    • MUST_UPDATE
    • UPDATE
    • MUST_DELETE
    • DELETE
    • NOTHING
For more details please refer to Using BIC to Update and Delete from the Database

The default behavior of BIC when writing to a database is:

After the record/segment is completed, it is written to the database using an SQL statement (INSERT). This process creates a new record in the table. If there already is a record with the same primary key, BIC triggers an error and the mapping fails!
However, you can modify the default behavior of BIC as follows:

COPY "INSERT" to 100:BICSQLMODE;

Each segment type/table has an internal field called BICSQLMODE. This field does not exist in the table. It is a control field used to instruct BIC to change the mode for that table. The following modes are available:

Mode

Description

INSERT(default)

Executes an insert (new record) for each instance of the segment. Duplicate keys will trigger an error.

INSERT_OR_UPDATE

Executes an insert (new record) for each segment. If the record already exists in the table (duplicate primary key), an UPDATE will be executed.

INSERT_DELETING

Executes an insert (new record) for each segment. If the record already exists in the table (duplicate primary key), it will be deleted first.

MUST _UPDATE

Updates the record in the table. If the record does not exist within the table, an error will occur.

UPDATE

Updates the record in the table. It does not check if the record exists.

MUST_DELETE

Deletes the record from the table. If the record does not exist in the table, an error will occur.

DELETE

Deletes the record from the table.

NOTHING

Leaves the record unchanged.

Refer to the following example for the different modes:
Please assume you receive a file with articles (e.g. PRICAT). In the file, there is a field called TYPE that indicates what to do (create, delete, or modify the price). If you receive a modification, you only want to update the price.

The following mapping can be used:

if IN:TYPE= "NEW"
  COPY "INSERT" to OUT:BICSQLMODE ;

  COPY IN:EAN TO OUT:EAN;
  COPY IN:QTY TO OUT:QTY;
  COPY IN:VAT TO OUT:VAT;
  COPY IN:PRICE TO OUT:PRICE;
  COPY IN:DESC TO OUT:DESC;
  COPY IN:INTERNALCODE TO OUT:INTERNALCODE;

endif

if IN:TYPE= "DEL"
 COPY "DELETE" to OUT:BICSQLMODE; 

 COPY IN:EAN TO OUT:EAN;
endif


if IN:TYPE= "MODIFY_PRICE"
  COPY "UPDATE" to OUT:BICSQLMODE;

  // we only update the price

  COPY IN:EAN TO OUT:EAN; 
  COPY IN:PRICE TO OUT:PRICE;
Endif

// now force writing to the database
flushSQLDestinationTables("OUT","on");

Note:
When applying DELETE, set the value of the primary key (in the example: EAN) to specify the record that has to be deleted.
When applying UPDATE, set the values of the primary key (EAN) and the fields that have to be updated (in the example: PRICE).

For the correct use of the different modes, you must understand the concept of primary key.

Attention: Deleting and updating from the database is a dangerous activity, because a wrong mapping can delete valuable data from the tables. Therefore, BIC never deletes/updates more than one record simultaneously. This will prevent deletion of the complete table, if the primary key is not properly defined.
