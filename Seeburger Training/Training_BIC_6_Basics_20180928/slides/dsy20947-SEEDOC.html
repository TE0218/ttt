Multireader / Multiwriter

  • It is possible to read from different files simultaneously and also to write to different files at once. However there is no GUI available.

  • There are a couple of commands related to this feature:

    • SetDocReader

    • SwitchDocWriter

    • SwitchDocReader

    • getDocWriterName

    • getDocReaderName

    • setSQLDocWriter

    • SetDocWriter

    • setAS400DocWriter
For more details please refer to Usage of Multiple Doc Readers and Writers

It is possible to create different reading and writing sources, different readers and writers within a single mapping, and to switch between them during conversion. This is an advanced and powerfull feature, but It has limited support in the BIC Mapping Designer user interface. The use of drag-and-drop or input completion does only work for the primary input and output formats which are visible. Especially note the following:

  1. The creation of new files, stream sources, or destinations (so-called doc reader and doc writer) and the switching between them can only be managed with the use of mapping commands.
  2. A single mapping can contain only record mapping programs for a single source message description. Therefore the usage of multiple readers has the following constraints:
    • All doc readers must use the same message type and the same message description.
    • All identical segments/records of all doc readers are calling the same record mapping program in the mapping. It does not matter which doc reader is currently in use. Take care about the "correct use" and the management of the different doc readers in the mapping code of that record mapping program.
  3. There are no such constraints for doc writers. It is possible to create any possible different output file or stream using different message types and different message descriptions (e.g. simultaneous use of XML and EDIFACT).
  4. The graphical representation of a mapping in the BIC Mapping Designer still contains only the main/default doc reader and writer. It does not matter if you use and access different doc readers/writers in the mapping. A validation of the addresses used in the mapping code is done against the default doc reader and writer only. The usage of a doc writer of a different type or message descriptions may result in compilation warnings (compiler version 2). The compiler version 1 does not check addresses at all, so it should be possible to compile the mapping without any further problem.
  5. The integration in BIS 6 allows the creation of multiple destination message attachments. If your doc writer should create a new destination attachment, please use the word "automatically" as file name.
  6. In XI 1.5, 1.6, 1.7 or 2.0, 2.1,2.2 it is NOT possible to access multiple attachments with multiple doc readers from within a mapping.
  7. For BIC in BIC MD and BIS6 it is possible to have multiple sources by also using the file name "automatically" for the setDocReader.
  8. In the BIS 6 and SAP XI environment conversions are managed by the runtime, the use of direct file access (by name) or re-use of input streams is not possible.
  9. Configuration commands (setSourceDelimiter, etc.) work on the current doc reader/writer. You have to call them after the doc reader/ writer is initialized and switched. These settings are only valid for the currently activated doc reader/writer. If you switch the doc reader, the changes take effect after the current mapping record program is finished, and the next record will be read from the new active reader.
  10. Whenever a record mapping program is finished, BIC continues with reading on the current activated doc reader. To continue with reading in another doc reader, please switch to that reader.
  11. Every doc reader and writer does its own syntax check. Each syntax check does not affect the others.
  12. Global procedures have a special behavior: they are initialized with the default doc reader and writer (main source/destination stream). Global procedures are not affected by switchDocReder/Writer calls outside in the mapping. If the doc reader/writer should be switched within a global procedure, the switchDocRedaer/writer command has to be executed there, too. This also does not affext the doc reader/writer settings of the main mapping.
